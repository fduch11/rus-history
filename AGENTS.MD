# Постоянные правила автономной разработки

Этот файл обязателен для чтения перед любыми изменениями репозитория `fduch11/rus-history`.

## Порядок работы

1. Открыть репозиторий `fduch11/rus-history`.
2. Сначала полностью прочитать `ROADMAP.md`.
3. Проверить, остались ли незавершённые пункты.

### Если все пункты завершены

- выполнить итоговый аудит проекта;
- проверить работоспособность приложения;
- сформировать один итоговый отчёт;
- после этого больше не изменять репозиторий.

### Если остались незавершённые пункты

1. Самостоятельно выбрать следующий логически завершённый этап из `ROADMAP.md`.
2. Реализовать только выбранный этап.
3. Не переходить к следующему этапу в той же итерации.
4. Сохранять работоспособность приложения после каждого изменения.
5. Не запрашивать подтверждение перед выполнением выбранного этапа.

## Обязательные проверки перед коммитом

Перед каждым коммитом необходимо:

- проверить синтаксис всех JavaScript-файлов;
- проверить существование всех локальных путей из HTML, CSS, JavaScript, manifest и service worker;
- убедиться, что приложение открывается локально через `index.html` без сервера;
- убедиться в отсутствии сетевых и внешних зависимостей;
- проверить, что ранее работающие функции не сломаны.

## Технологические ограничения

Разрешено использовать только:

- HTML;
- CSS;
- чистый JavaScript;
- локальные файлы проекта;
- `localStorage` и стандартные браузерные API, не требующие внешней сети.

Запрещено использовать:

- npm;
- React;
- Vue;
- другие фреймворки и внешние библиотеки;
- CDN;
- внешние API;
- `fetch`;
- сервер, необходимый для базовой работы приложения;
- зависимости, без которых приложение не открывается локально.

## ROADMAP.md

После завершения каждого этапа необходимо обновить `ROADMAP.md`:

- отметить выполненные пункты;
- описать результат этапа;
- указать один следующий этап;
- обновить журнал итераций.

## Коммиты

- Работать напрямую в ветке `main`.
- За одну итерацию делать один логически завершённый этап.
- Все изменения этапа и обновление `ROADMAP.md` включать в один коммит.
- Делать ровно один коммит на итерацию.
- Использовать конкретное и понятное сообщение коммита.

## Итоговый ответ после итерации

Вернуть краткий отчёт, содержащий:

- какой этап выполнен;
- какие файлы изменены;
- какие проверки выполнены;
- хеш и сообщение коммита;
- какой этап указан следующим.

## Приоритет правил

При каждом новом запуске сначала применять этот файл, затем `ROADMAP.md`. Если правила расходятся, действовать по более строгому ограничению, сохраняя локальную работоспособность приложения и запрет на внешние зависимости.
